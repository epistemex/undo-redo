Dereferencing Objects

One way undo-redo allows for flexibility is to store provided objects AS-IS.

With this flexibility you can store almost any kind of objects, current and future without being limited to any built-in form of serializing and deserializing.

This of course transfers the responsibility to the developer to serialize and dereference objects for instance when this is needed.

To understand object references a little better take a look at this post.

A simple way to serialize objects is to use JSON:

 stack.add(JSON.stringify(obj));

and when the data is needed again deserialize:

const obj = stack.canUndo() ? JSON.parse(stack.undo()) : null;
const obj = stack.canRedo() ? JSON.parse(stack.redo()) : null;

For objects such as TypedArrays you have to do a copy. A fast way to copy is:

stack.add(typedarray.slice(0));

and the same when retrieving.

For things like canvas bitmap, which undo-redo was originally designed for, you can store and restore using getImageData() and putImageData():

 stack.add(context.getImageData(0, 0 , w, h));

 const bitmap = stack.undo(); 
 if (bitmap) {  
   context.putImageData(bitmap, 0, 0);
 }

Since getImageData() already provide a copy of the content of the canvas there is no need to serialize or deserialize here unless you work on that bitmap directly, which then require a TypedArray copying.
